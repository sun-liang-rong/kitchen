# 厨房许愿池原型与需求匹配检测

## 1. 最新检测结论

本文件已根据最新原型更新。当前 `prototype/index.html` 已补齐上一轮 P0 缺口，原型已经从“概念高保真原型”升级为“可走通 MVP 核心闭环的业务原型”。

本轮已补齐：

- 双人绑定/已绑定状态展示。
- 厨房状态可设置并同步。
- 发愿页增加希望时间。
- 发愿页补充更多感觉、替代和搭把手选项。
- 提交愿望后真实新增愿望卡。
- 回应弹层增加结构化原因和改天时间。
- 回应后真实更新愿望卡状态。
- 兑现弹层增加实现人、搭手完成情况、加入常吃。
- 兑现后新增记录，并把菜加入“我们家的菜”。

当前仍可继续优化的 P1/P2 项：

- 菜库增加新增/编辑入口和筛选。
- 发愿时根据对方厨房状态给更明显的提醒。
- 通知/消息中心。
- 未绑定、空许愿池、空菜库、空记录等空状态。

## 2. 本轮补充记录

本轮继续补齐了上一轮剩余的关键 P1 流程：

- 许愿池增加状态筛选：池中、待确认、已认领、改天、已兑现。
- 新增愿望详情页。
- 详情页展示愿望完整信息、状态流转、回应历史。
- 回应“换个版本/做轻松版/改天实现”后进入“待她确认”。
- 详情页支持模拟对方确认。
- 对方确认后，愿望从“待她确认”变为“已认领”，并回到可兑现状态。

当前原型已经具备比较完整的 MVP 业务路径：

```text
许愿 -> 回应 -> 等待确认 -> 对方确认 -> 兑现 -> 加入常吃/记录历史
```

## 3. 首轮检测结论归档

当前原型与需求方向整体匹配，已经表达出“厨房许愿池”的核心产品气质：

- 用“许愿”替代“点菜”。
- 双方都能看到共享许愿池。
- 愿望卡包含许愿人、强度、搭把手、状态等关键概念。
- 支持发起许愿、回应愿望、记录兑现、查看我们家的菜。
- 文案整体温柔，没有使用“拒绝、订单失败、任务取消”等生硬表达。

但当前原型仍属于“概念高保真原型”，还没有完整覆盖 MVP 业务需求。主要缺口集中在：

- 双人绑定关系未体现。
- 厨房状态只能看，不能设置。
- 许愿字段不完整。
- 愿望回应缺少结构化原因和完整改期选择。
- 愿望状态没有真实流转。
- 兑现记录缺少“谁实现、谁搭手、是否加入常吃”等关键字段。
- 我们家的菜还缺少新增、编辑、从兑现转入的路径。

## 4. 首轮逐项检测归档

## 4.1 核心业务闭环

需求闭环：

1. 进入 app，看到共享许愿池。
2. 任意一方发起愿望。
3. 选择愿望强度、可接受替代、愿意搭把手。
4. 另一方回应：认领、换轻松版、改到周末、一起做、先搁着。
5. 状态进入对应阶段。
6. 饭后记录兑现。
7. 沉淀愿望历史、常吃菜和口味偏好。

当前原型：

- 第 1、2、3、4、6 步已基本体现。
- 第 5 步只在静态卡片上体现，没有交互后的真实状态变化。
- 第 7 步有“我们家的菜”和“兑现记录”，但没有从兑现结果转入常吃菜的真实操作。

结论：核心闭环概念成立，但还需要补足动态状态变化。

## 4.2 许愿池首页

已匹配：

- 展示当前愿望卡。
- 展示我许的、她许的。
- 展示已认领、改天实现、先搁着等状态。
- 展示今日厨房状态。
- 有快捷操作“我来回应”“周末安排”。

缺口：

- 首页没有单独筛选“已认领、改天实现、已兑现”。
- 愿望卡没有明确展示“希望时间”字段。
- 愿望卡没有明确展示“创建时间”。
- 愿望卡没有统一展示“最近回应”。
- “已兑现愿望”只在记录页体现，首页许愿池没有体现。

建议：

- 增加状态筛选：池中、已认领、周末、已兑现。
- 愿望卡底部增加一行轻量信息：希望时间、创建时间、最近回应。

## 4.3 发起许愿

已匹配：

- 可以输入想吃什么。
- 可以选择想要的感觉。
- 可以选择愿望强度。
- 可以选择可接受替代。
- 可以选择愿意搭把手。
- 提交后回到许愿池并显示提示。

缺口：

- 缺少“希望时间”：今晚、明天、这周、周末、有空再说。
- “想要的感觉”选项不完整，缺少有肉、不想吃米饭、想吃面、想吃上次那个。
- “可接受替代”缺少“家里有什么就做什么”。
- “搭把手”缺少洗菜、收拾厨房、晾衣服、饭后收桌、陪你一起做、给你买杯喝的、今天简单吃也可以。
- 提交后没有真实新增一张愿望卡。

建议：

- 发愿页增加“希望时间”分组。
- 补全搭把手选项。
- 原型层面可以在提交后新增一张临时愿望卡，增强闭环感。

## 4.4 认领愿望

已匹配：

- 有回应弹层。
- 有今晚实现、做轻松版、换个版本、周末安排、一起做、先放池子里。
- 有原因/提议输入框。
- 文案较柔和，没有使用“拒绝”。

缺口：

- “改天实现”只体现为周末安排，缺少明天、这周、指定日期、有空再做。
- 回应原因没有结构化选项。
- “换个版本”目前写死为“换成红烧鸡腿”，缺少通用换菜入口。
- 发送回应后卡片状态没有更新。
- 没有体现双方确认的过程。

建议：

- 回应弹层增加“原因标签”。
- 改天实现增加时间选择。
- 发送后将原卡片状态更新为“换个版本”或“已认领”。

## 4.5 厨房状态

已匹配：

- 首页显示双方厨房状态。
- 能体现“她有点累”的业务语境。

缺口：

- 不能设置自己的厨房状态。
- 缺少状态选项：想认真做一顿、正常、有点累、只想简单吃、不想开火、适合一起做。
- 发愿时没有根据对方状态给提示。
- 回应时没有根据自己状态给建议。

建议：

- 首页状态卡点击后打开“设置今日厨房状态”弹层。
- 发愿页顶部增加状态提醒：对方今天有点累，建议许愿快手菜或一起完成。

## 3.6 愿望兑现

已匹配：

- 有兑现弹层。
- 可以填写实际吃了什么。
- 有饭后反馈标签。
- 不使用星级评分，符合需求原则。
- 兑现后进入记录页。

缺口：

- 缺少“谁实现的”。
- 缺少“谁搭了什么手”。
- 缺少“是否加入常吃”的明确开关。
- 反馈选项不完整，缺少今天很好吃、有点淡、有点辣、不够辣、太麻烦下次周末做等。
- 兑现后没有真实新增记录，也没有转入我们家的菜。

建议：

- 兑现弹层增加实现人、搭把手完成情况、加入常吃开关。
- 点击兑现后记录页新增本次记录。

## 3.7 我们家的菜

已匹配：

- 展示菜名。
- 展示谁会做。
- 展示适合时间和难度。
- 展示常吃、愿望兑现过等标签。

缺口：

- 不能新增菜。
- 不能编辑菜。
- 没有从愿望兑现转入菜库的操作。
- 没有按工作日、周末、快手、谁会做等维度筛选。

建议：

- 增加“添加一道菜”入口。
- 从兑现弹层勾选“加入常吃”后，在菜库中体现。

## 4. 优先修改建议

### P0：必须补

1. 增加双人绑定/已绑定状态入口。
2. 发愿页增加“希望时间”。
3. 厨房状态支持设置。
4. 回应后更新愿望卡状态。
5. 兑现弹层补充“谁实现、谁搭手、是否加入常吃”。

### P1：应该补

1. 许愿池增加按状态筛选。
2. 发愿页补全搭把手选项。
3. 回应弹层增加结构化原因。
4. 改天实现增加明天、这周、指定日期。
5. 兑现后新增记录，并同步到菜库。

### P2：体验优化

1. 首页增加“今天想往厨房许愿池里丢点什么？”作为更强的第一屏文案。
2. 愿望卡展示希望时间和最近回应。
3. 菜库增加筛选：快手、周末、我会做、她会做、常吃。
4. 空状态补齐，例如没有愿望时引导“许个愿”。

## 5. 最终判断

当前原型与产品方向匹配，适合用于早期沟通“厨房许愿池是什么”。

如果要作为开发前的完整 MVP 原型，还需要补足业务字段、状态流转和双人绑定流程。

建议下一步先改 P0 项，让原型从“概念展示”升级为“完整 MVP 流程原型”。
