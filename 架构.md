# 厨房许愿池项目架构需求

## 1. 项目概述

**厨房许愿池**是一款面向情侣、夫妻、双人小家庭的移动端 app。

产品核心是：双方都可以把“想吃什么”投进共享许愿池，也都可以认领对方的愿望，通过实现、换菜、延期、一起做、先搁置等方式完成一次吃饭协商。

本项目采用：

- 前端：Flutter
- 后端：NestJS
- 数据库：PostgreSQL
- 缓存：Redis
- ORM：Prisma
- 架构形态：模块化单体

第一版目标是完成“许愿 -> 回应 -> 兑现 -> 沉淀常吃菜”的最小业务闭环。

## 3. 架构原则

1. 第一版优先完成核心闭环，不做过度工程。
2. 后端采用模块化单体，避免一开始拆微服务。
3. 前端采用清晰的 feature 分层，方便后续扩展页面。
4. 所有业务状态以服务端数据为准，前端只做本地缓存和临时展示。
5. 接口先使用 REST API，后期需要实时协作时再增加 WebSocket。
6. 推送、图片、AI 推荐等能力预留接口，但第一版只做必要功能。

## 6. 后端模块设计

### 6.1 AuthModule

负责用户登录、注册、刷新 token、退出登录。

第一版可支持：

- 手机号 + 验证码登录
- 邮箱 + 密码登录

如果想降低开发成本，第一版可以先做邮箱 + 密码登录，手机号验证码后续接入短信服务。

核心接口：

- 注册
- 登录
- 刷新 token
- 退出登录
- 获取当前用户

### 6.2 UsersModule

负责用户资料。

用户字段：

- 昵称
- 头像
- 性别，可选
- 生日，可选
- 默认厨房状态

核心接口：

- 获取我的信息
- 修改我的信息
- 上传头像

### 6.3 CouplesModule

负责两人关系绑定。

第一版只支持一对一关系。

绑定方式建议：

- A 生成邀请码
- B 输入邀请码
- 双方绑定成功

核心接口：

- 生成绑定邀请码
- 输入邀请码绑定
- 获取当前伴侣关系
- 解除绑定，后续版本再做

### 6.4 WishesModule

负责许愿卡主体。

核心能力：

- 创建愿望
- 获取许愿池列表
- 获取愿望详情
- 修改愿望
- 删除自己的愿望
- 更新愿望状态

愿望类型：

- 具体菜名
- 想要的感觉

愿望状态：

- 池中
- 已认领
- 换个版本
- 改天实现
- 一起完成
- 先搁着
- 已兑现

### 6.5 WishResponsesModule

负责对愿望的回应。

回应类型：

- 今晚实现
- 做轻松版
- 换个版本
- 改天实现
- 一起完成
- 先搁着

回应原因：

- 今天有点累
- 家里没食材
- 时间不够
- 这道菜太麻烦
- 今天想吃清淡点
- 可以做，但需要帮忙
- 周末更适合认真做
- 今天不想开火

### 6.6 WishFulfillmentsModule

负责饭后兑现记录。

记录内容：

- 实现了哪个愿望
- 谁实现的
- 谁搭了什么手
- 实际吃了什么
- 反馈标签
- 是否加入常吃
- 备注

### 6.7 KitchenStatusModule

负责每日厨房状态。

状态选项：

- 今天想认真做一顿
- 今天正常
- 今天有点累
- 今天只想简单吃
- 今天不想开火
- 今天适合一起做

核心接口：

- 设置今日厨房状态
- 获取我和对方今日状态
- 获取最近状态记录，后续版本

### 6.8 DishesModule

负责“我们家的菜”。

菜品来源：

- 用户手动添加
- 已兑现愿望转入
- 高频愿望推荐转入，后续版本

菜品字段：

- 菜名
- 谁会做
- 适合时间
- 难度
- 标签
- 最近一次反馈
- 是否常吃

### 6.9 NotificationsModule

负责通知。

第一版先记录站内通知，后续再接入系统推送。

通知场景：

- 对方许了新愿望
- 对方回应了我的愿望
- 对方认领了愿望
- 愿望被改天实现
- 愿望已兑现

### 6.10 UploadModule

负责头像、菜品图片等文件上传。

第一版可以只支持头像上传，菜品图片后续再做。

## 8. API 设计原则

1. 使用 REST API。
2. 所有接口统一返回结构。
3. 所有需要登录的接口必须携带 access token。
4. 所有涉及情侣数据的接口必须校验用户是否属于当前 couple。
5. 业务状态变更必须在服务端校验，前端不能直接随意改状态。

### 8.1 统一返回格式

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

错误示例：

```json
{
  "code": 40001,
  "message": "愿望不存在",
  "data": null
}
```

## 9. 鉴权与安全

### 9.1 鉴权方式

使用 JWT：

- access_token：短有效期，例如 2 小时。
- refresh_token：长有效期，例如 30 天。

移动端保存：

- access_token 和 refresh_token 使用 flutter_secure_storage。

### 9.2 权限规则

1. 用户只能访问自己的数据。
2. 用户只能访问当前情侣关系内的数据。
3. 愿望只能由创建者修改内容。
4. 愿望回应可以由对方创建。
5. 兑现记录可以由双方创建或补充，具体规则第一版可放宽。

### 9.3 安全要求

- 密码必须哈希存储，使用 bcrypt 或 argon2。
- 所有接口必须做参数校验。
- 服务端必须校验愿望状态流转是否合法。
- 敏感配置必须放入环境变量。
- 生产环境必须使用 HTTPS。

## 20. 技术决策总结

第一版推荐技术组合：

```text
App:
Flutter + Riverpod + go_router + dio + freezed + Hive

Backend:
NestJS + Prisma + PostgreSQL + Redis + JWT + Swagger

Infra:
Docker Compose + PostgreSQL + Redis + OSS/S3 + GitHub Actions

Push:
第一版站内通知
后续接入 APNs / FCM / 国内推送服务
```

该架构的重点是：先用简单稳定的模块化单体把业务闭环跑通，再根据真实使用情况逐步增加实时同步、推送、AI 推荐和图片能力。
