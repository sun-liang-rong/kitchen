# 厨房许愿池功能执行清单

本清单按 MVP 闭环拆分：每一项先完成后端，再完成前端，最后自测通过后进入下一项。

## 1. 需求与执行边界

- 前端：确认页面范围包含登录、注册、绑定、许愿池、发愿、详情、回应、兑现、我们家的菜、记录、我的页面。
- 后端：确认接口范围覆盖认证、用户资料、伴侣关系、厨房状态、愿望、回应、确认、兑现、菜库。
- 自测：清单文件存在，功能点均包含前端、后端、测试标准。

## 2. 登录注册与双人绑定

- 前端：实现登录页、注册页、绑定引导页、输入邀请码页、绑定申请确认页、已绑定状态页和解绑确认。
- 后端：提供注册、登录、登出、获取当前用户、生成邀请码、通过邀请码发起绑定、同意、拒绝、取消、解绑接口。
- 自测：新用户可注册登录；两名用户可通过邀请码完成绑定；已绑定用户不能重复绑定；解绑后不再共享新的许愿池数据。

## 3. 后端 MVP 接口

- 前端：了解 REST 数据结构，后续页面统一使用相同枚举和字段。
- 后端：补齐核心 controller/service/DTO，支持内存 MVP 数据流转，避免数据库未启动时阻断开发。
- 自测：`pnpm build` 通过；核心 service 单元测试通过。

## 4. 前端领域模型与应用状态

- 前端：建立愿望、回应、兑现、厨房状态、常吃菜、伴侣关系模型和 Riverpod 状态管理。
- 后端：保持字段与前端模型一致。
- 自测：`flutter analyze` 通过；状态管理单元测试通过。

## 5. 许愿池首页与发起许愿

- 前端：实现共享许愿池首页、状态筛选、厨房状态设置、发愿表单、提交后新增愿望卡。
- 后端：提供愿望列表、创建愿望、厨房状态读写接口。
- 自测：新增愿望后首页出现卡片；状态筛选正确；厨房状态可修改。

## 6. 愿望详情、回应与确认

- 前端：实现愿望详情、完整字段展示、回应弹层、结构化原因、改天时间、待确认和确认流程。
- 后端：提供创建回应、确认回应、状态流转接口。
- 自测：回应后愿望状态正确变化；需要确认的回应可被确认后进入已认领。

## 7. 饭后兑现、记录与我们家的菜

- 前端：实现兑现表单、实现人、搭手完成、反馈标签、加入常吃、记录页、菜库新增/编辑/筛选。
- 后端：提供兑现接口、记录查询、菜库增改查接口；兑现时可自动加入菜库。
- 自测：兑现后记录新增；勾选加入常吃后菜库出现对应菜；已兑现愿望不再作为待处理愿望。

## 8. 最终回归

- 前端：完成一次完整路径：注册 -> 登录 -> 双人绑定 -> 许愿 -> 回应 -> 确认 -> 兑现 -> 加入常吃。
- 后端：构建和测试全部通过。
- 自测：运行后端测试、后端构建、Flutter analyze、Flutter test。
